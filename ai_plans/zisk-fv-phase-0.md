# Phase 0 detailed plan — ZisK FV extractor spike

**Parent metaplan.** `ai_plans/zisk-fv-metaplan.md` (Phase 0 section).

## Context

Phase 0's single purpose is to burn down the highest-risk unknown in the ZisK FV metaplan: can PIL2 constraints from `pil/zisk.pilout` be extracted into Lean in a shape compatible with per-opcode reasoning, following the pattern of `openvm-fv`? Three things change how concretely we can answer this, all confirmed by parallel reconnaissance:

1. **`zisk.pilout` is structured and walkable.** It's a protobuf file (schema at `/home/cody/pil2-proofman/pilout/src/pilout.proto`) exposed via a clean Rust API `PilOutProxy::new(path)` from the `pilout` crate. There's prior art in `/home/cody/pil2-proofman/cli/src/commands/pil_helpers.rs` that walks pilout and emits Rust via TinyTemplate — a direct template for our Lean emitter.
2. **The openvm-fv extractor exists publicly** (`openvm-fv/extractor/` on `origin/develop-v2.0.0`, ~60 lines of Rust) but it consumes Plonky3 `SymbolicConstraintsDag`, not PIL2 pilout, so we cannot reuse the extractor itself. We *do* reuse its output shape and the `Circuit` typeclass from `NethermindEth/leanzkcircuit`.
3. **RV64 ADD's lifecycle is fully mapped.** RV64 `ADD` → Zisk op `0x0a`, `OpType::Binary`, routes via `OPERATION_BUS_ID=5000` (11-field schema) to `state-machines/binary/pil/binary_add.pil`'s `BinaryAdd` AIR, which proves the carry-chain arithmetic on 16-bit limbs.

Phase 0 will extract **one constraint** from the `BinaryAdd` AIR, prove a trivial lemma about it, and ship the extractor + Lean scaffold that every subsequent phase builds on.

## Scope (strict)

**In scope:**
- A Rust CLI tool at `tools/zisk-pil-extract/` that, given `zisk.pilout` and an AIR name, emits one `.lean` file containing the AIR's constraints in a shape matching `openvm-fv/OpenvmFv/Extraction/*.lean`.
- A Lean 4 package at `ZiskFv/` (sibling to `openvm-fv/`) with minimal dependencies: Mathlib, `NethermindEth/leanzkcircuit`, `NethermindEth/sail-riscv-lean` (branch `main`, for `LeanRV64D`).
- Minimal `ZiskFv.Circuit` typeclass analogue sufficient to make the extracted file typecheck. If `LeanZKCircuit.OpenVM.Circuit` already works as-is (it's field-polymorphic), we use it unchanged; otherwise we define a thin shim.
- Extract the single simplest meaningful constraint from `BinaryAdd`: a boolean flag constraint of the form `cout[i] * (1 - cout[i]) === 0` (source: `state-machines/binary/pil/binary_add.pil:14`).
- Prove one trivial lemma in Lean: `cout[0]` is boolean. Pipeline exercise, not semantics.
- `justfile` with `just verify-phase0` target that builds the extractor, runs it on `zisk.pilout`, runs `lake build`, and diffs against a hand-translated reference.
- `docs/fv/extractor-notes.md` capturing the extractor contract, observed pilout corner cases, and known limitations.
- `docs/fv/phase-0-handoff.md` capturing what shipped, what was learned, what Phase 1 should recalibrate.
- Update `MEMORY.md` and memory files with Phase 0 outcomes.

**Explicitly out of scope:**
- Full Main AIR extraction (Phase 1).
- Operation-bus bridging between Main and secondary SMs (Phase 1 — this is the hard compositional-proof work).
- Any semantic proof about ADD (Phase 1).
- Transpiler axiomatization (Phase 1).
- Golden-trace harness (Phase 1).
- Any opcode beyond the single boolean constraint (Phase 1+).

## Execution order

Serial. Each step's output is the next step's input.

1. **Read the openvm-fv extractor** (read-only). `git show origin/develop-v2.0.0:extractor/src/main.rs` in the openvm-fv tree. Confirms emission style.
2. **Read pil-helpers template.** `/home/cody/pil2-proofman/cli/src/commands/pil_helpers.rs` + linked TinyTemplate. This is the walker pattern we copy.
3. **Probe `zisk.pilout` structure.** Run the existing `pil2-proofman pilout inspect --pilout pil/zisk.pilout` to confirm airgroup/AIR names. Locate `BinaryAdd` by substring match. Capture its `num_rows`, constraint count, and symbol table.
4. **Scaffold `tools/zisk-pil-extract/`.** `Cargo.toml` depending on `pilout` via path to `/home/cody/pil2-proofman/pilout` (git dep to upstream later). `src/main.rs` with clap CLI: `zisk-pil-extract --pilout <path> --air BinaryAdd --output ZiskFv/ZiskFv/Extraction/BinaryAdd.lean`.
5. **Implement minimal extraction.** For the target AIR:
   - Walk `air.symbols` to build a column index → name map (for readable comments / hand-comparison).
   - Walk `air.constraints` and for each constraint, walk its expression tree (`air.expressions[constraint.expression_idx]`) recursively, emitting Lean syntax of the form `(Circuit.main c (id := <airgroup>) (column := <col>) (row := row) (rotation := <rot>)) + ... = 0`.
   - Start with enough expression-tree cases to handle our target constraint (boolean: one mul, one sub, one constant). Fail loudly on unhandled cases.
6. **Hand-translate the target constraint** into `ZiskFv/ZiskFv/Extraction/BinaryAdd.hand.lean`. Used as an oracle.
7. **Scaffold `ZiskFv/` Lean package.** `lakefile.toml` mirroring `openvm-fv`'s. Add the extraction output file as an import. Run `lake build` and confirm typecheck.
   - If `LeanZKCircuit.OpenVM.Circuit` doesn't typecheck under Goldilocks (e.g. it fixes `F = FBB`), add a thin `ZiskFv.Circuit` shim generalizing over `F` as Goldilocks.
   - Add `ZiskFv/Fundamentals/Goldilocks.lean` following the shape of `openvm-fv/OpenvmFv/Fundamentals/BabyBear.lean` (field, prime, `Field` instance).
8. **Prove the trivial lemma** in `ZiskFv/ZiskFv/Spike.lean`: `lemma cout_0_boolean : ∀ (c : Valid_BinaryAdd FGL) (row : ℕ), c.cout_0 row 0 = 0 ∨ c.cout_0 row 0 = 1`, using the extracted constraint. Tactic: unfold + `grind`. Confirm `lake build` green.
9. **Diff harness.** A trivial shell test or Rust test that runs `zisk-pil-extract` and diffs the output against `BinaryAdd.hand.lean` (normalizing whitespace / alpha-equivalence). Must pass.
10. **Write `justfile`** with `verify-phase0`:
    ```
    verify-phase0:
        cargo run --manifest-path tools/zisk-pil-extract/Cargo.toml -- \
            --pilout pil/zisk.pilout --air BinaryAdd \
            --output ZiskFv/ZiskFv/Extraction/BinaryAdd.lean
        diff -w ZiskFv/ZiskFv/Extraction/BinaryAdd.lean ZiskFv/ZiskFv/Extraction/BinaryAdd.hand.lean
        cd ZiskFv && lake build
    ```
11. **Write docs.** `docs/fv/extractor-notes.md` (extractor contract; pilout structure observations; limitations — what expression kinds / constraint types are not yet handled) and `docs/fv/phase-0-handoff.md` (≤500 words, for Phase 1 agent).
12. **Update memory.** Add a `project_phase0_outcomes.md` memory capturing anything learned that contradicts the metaplan. Update `MEMORY.md` index.

## Invariant (verification gate)

Phase 0 is complete iff `just verify-phase0` exits 0 from a clean checkout. This runs the extractor, diffs its output against the hand-written reference (must match), and compiles the Lean package including the trivial lemma (must typecheck with zero `sorry`).

## Known fragility (and what to do)

- **`LeanZKCircuit.OpenVM.Circuit` might be Plonky3-specific.** If so, we write a minimal `ZiskFv.Circuit` mirroring its interface but agnostic to the proof backend. Time box: 2 hours before pivoting to a shim.
- **pilout expression tree may have constructs we don't handle.** The target constraint needs only constants, mul, sub, column-ref. If the AIR's other constraints use challenges / extension-field ops / public inputs, we leave them unimplemented (with a clear error) and note in extractor-notes.md. Phase 1 fills in as needed.
- **`BinaryAdd` AIR may not exist by that exact name in pilout.** Agent 2's research was from `.pil` source; compiled pilout might use a different name (e.g. `Binary.Add`, `binary_add`, `binary/BinaryAdd`). The extractor uses substring matching to stay resilient. If the name is very different, update this plan before execution.
- **Lake dep resolution on a fresh tree may take a while.** `lake exe cache get!` is the openvm-fv mitigation and we mirror it.

## Decisions captured (reversible but preferred)

- Extractor lives **in-tree** at `tools/zisk-pil-extract/` rather than in a separate repo or a `pil2-proofman` fork. Easy to upstream later; keeps Phase 0 self-contained.
- Lean package lives at **`ZiskFv/`** (sibling to `openvm-fv/`). Mirrors the local reference repo's position.
- Dependencies use **git refs** (same style as openvm-fv's lakefile). Vendoring decision deferred to Phase 1 if coordination with Nethermind proves slow.

## Verification section (how we test end-to-end)

1. From clean checkout: `just verify-phase0`. Expected: exit 0.
2. Manual inspection: open `ZiskFv/ZiskFv/Extraction/BinaryAdd.lean` — confirm it contains a `def constraint_N` of the boolean form.
3. Manual inspection: open `ZiskFv/ZiskFv/Spike.lean` — confirm the lemma name / statement, confirm no `sorry`.
4. Re-run from a fresh shell: `cd ZiskFv && lake build` — must still be green (tests incrementalism).
5. Read `docs/fv/phase-0-handoff.md` — confirm it lists any metaplan revisions needed.
