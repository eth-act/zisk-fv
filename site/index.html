the question

What are you trusting when a ZisK proof verifies?

A ZKVM proof says: "this RISC-V program ran correctly on this circuit." But the circuit is just polynomial constraints; "ran correctly" means "matches the RISC-V architectural spec." Without a formal proof tying the two together, the claim is a wish. zisk-fv is that proof — and this page lays out the small set of axioms it ultimately rests on.

What we trust, exactly

Every per-opcode equivalence theorem ultimately depends on a small set of clearly-named axioms. They group into four categories. Anything outside this list is proven from scratch or follows from these.

Trusted spec The RISC-V specification itself — 1 source

We don't prove that the Sail-RISC-V model is correct — that's an inherited trust from the formally-developed Sail spec used by Nethermind / OpenLabs and the broader community. Every opcode's "intended behavior" is what the Sail model says it is. If you don't trust Sail, no per-opcode proof is meaningful.

SAIL
NethermindEth/sail-riscv-lean :: LeanRV64D
The translated Sail-RV64 architectural model. ZisK's RV64IM behavior must match this execute_instruction evaluation.

Trusted contract RISC-V → Zisk transpiler — ~63 axioms + 4 platform + 3 PC bridges

ZisK doesn't run RV64 instructions directly. It compiles them into Zisk microinstructions via core/src/riscv2zisk_context.rs (a Rust program). We axiomatize that translation — for each RV opcode, "if Main's columns are configured this way, then the Zisk witness corresponds to this RV instruction." We do not prove the transpiler; we declare the contract it must keep. A Rust-side proof would close this gap.

~63 axioms
transpile_<OP>
One per RV opcode. Pins the Main-AIR mode flags (e.g. op, store_pc, m32, jmp_offset2) in terms of the RV instruction encoding.
P1–P4
PlatformScope.{pmpCheck,within_clint,pmaCheck,update_elp}
Platform-feature assertions: the deployed ZisK runtime has no PMP/PMA filtering, no CLINT region in the address space, no extended-pointer state mutation. Only retire if the deployment changes.
TP-JAL / TP-JALR / TP-AUIPC
transpile_PC_for_<OP>
For the three store_pc=1 opcodes, asserts that Sail's program counter equals Main's pc column at the row index. Bridges the runtime PC value, not just the encoding.

Trusted protocol Bus permutation soundness — 7 axioms

ZisK's circuit is a constellation of AIRs (Algebraic Intermediate Representations) glued together by lookup and permutation arguments over named "buses". Proving these arguments sound from first principles would mean formalizing the proving system — out of project scope. Instead, each bus's protocol is trusted via a single axiom that says "the rows on bus X match across providers and consumers." This is the same approach openvm-fv takes. Listed by bus ID:

bus 5000op
OperationBus.matches_entry
Operation bus connecting Main to secondary state machines (Arith, Binary, …). Multiplexed opcode dispatch.
bus 125lookup
bin_table_consumer_wf
Binary lookup table (BT-B). Per-byte switch semantics for AND / OR / XOR / LT / LTU / EQ / GT / LE / LEU / MIN(U) / MAX(U) / ADD / SUB / SEXT.
bus 124lookup
bin_ext_table_consumer_wf
Binary extension lookup table (BT-X). Per-byte shift semantics for SLL / SRL / SRA across 32- and 64-bit operands.
bus 10memory
memory_bus_register_write_perm_sound
MB-W. Register-write permutation soundness for store_pc = 0 opcodes (the standard rd-write path).
bus 10memory
memory_bus_register_write_perm_sound_store_pc
MB-W-PC. Companion to MB-W for store_pc = 1 (JAL / JALR / AUIPC). Same trust class, different gating.
bus 10memory
lookup_consumer_matches_provider_load
MB-L. Load-side permutation soundness: a memory-side bus row pairs with a Mem AIR row at as = 2.
bus 10memory
lookup_consumer_matches_provider_store
MB-S. Store-side permutation soundness, parallel to MB-L for the write path.

Trusted model fidelity Mem AIR ↔ Sail state.mem — 2 axioms

The bus protocol tells us that Mem AIR rows match memory-bus consumers; but it doesn't say what Mem AIR is in terms of the architectural memory. These two axioms close that gap by stating: a Mem AIR row that matches a memory-bus entry agrees with Sail's state.mem byte-by-byte. They sit at the boundary between ZisK's operational memory model and the Sail spec's logical memory.

MS-L
row_models_sail_state_load
Load: a Mem AIR row matching the load bus entry agrees with Sail's pre-state memory at the projected addresses.
MS-S
row_models_sail_state_store
Store: same shape but for post-state. Closes the store side of the implementation-models-spec faithfulness gap.

Everything else — ALU semantics, sign-extension boundaries, multiplicative carry chains, four-quadrant signed division overflow, byte-level arithmetic — is derived in Lean from these axioms plus Mathlib. No sorry stubs.

how the proof is structured

Three layers, two languages, one theorem shape

Each per-opcode proof is a small composition: take the Sail pure spec, take the named-column AIR layer, plug them into the right bus model, and discharge the residual via a per-archetype discharge lemma. The toolkits below are what made the discharge mechanical.

Spec side

RV64D/<op>.lean — pure Lean reformulation of each Sail-RV64 opcode (execute_instruction unfolded into a clean BitVec 64 function). Every spec file ships an equivalence lemma execute_<OP>_pure_correct proving it equals the unfolded Sail body.

Circuit side

Extraction/ renders zisk.pilout into raw Lean constraint structures. Airs/ wraps those into Valid_<AIR> records with named-column accessors and per-constraint bridge lemmas. The Main AIR multiplexes opcodes; secondary AIRs (Arith, Binary, Mem, MemAlign) handle heavy ops.

Composition

Spec/<Op>.lean proves: "if the relevant AIR rows validate and the bus carries the right entry, the Sail spec output equals the bus-effect output." Equivalence/<Op>.lean wraps that into the metaplan target shape, with a Tier-1 companion that closes the rd-value derivation entirely from circuit constraints.

Toolkits (Lean infrastructure)

PackedBitVec/{NoWrap, MulNoWrap, SignedNoWrap, WidePCNoWrap, Extensions, Signed}.lean — generic FGL ↔ BitVec 64 lifts handling additive carry chains, multiplicative chunks, four-quadrant signed witnesses, and PC-wide values that exceed the Goldilocks prime. None of these are trust items; all are proven.

a note on what could go wrong

What this doesn't prove

Honest scope. Three things this project deliberately doesn't cover, each with a reason.

The proving system itself

PLONK / plookup / logUp / permutation argument soundness as Lean theorems is out of scope. The bus-protocol axioms (BT-B, BT-X, MB-*) trust this layer at the bus-entry level. Closing it would mean formalizing the proving system itself.

The transpiler implementation

We axiomatize the contract that riscv2zisk_context.rs obeys, but we don't prove the Rust code matches it. A separate Rust-side verification effort would close this gap.

The Sail spec's correctness

The Sail RISC-V model is taken as ground truth. If a bug exists in Sail, this project inherits it. Sail is widely audited by the RISC-V community, so this is the conventional bar.
