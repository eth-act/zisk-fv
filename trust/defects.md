# Defect ledger

This ledger tracks known defects that affect the formal verification claim.
The ledger is not the trust ledger: do not add axioms to cover these rows.
Each open defect must either block the unqualified theorem or appear as an
explicit exception predicate in the theorem statement.

Every known defect must be visible in three places:

1. a human-readable ledger entry in this file;
2. a Lean predicate under `ZiskFv/Compliance/Defects.lean` naming the exact
   excluded behavior or blocked witness shape;
3. a top-level theorem statement whose name and hypotheses make the
   exception explicit.

The theorem must exclude the smallest behavior justified by the evidence.
Excluding an entire opcode is allowed only when the defect covers the whole
opcode. Ordinary out-of-scope items, such as precompiles or non-RV64IM
extensions, belong in scope documentation rather than this ledger.

| Kind                      | Meaning                                                                                                                    | Theorem treatment                                                                                                                                        |
|---------------------------|----------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------|
| `implementation-semantic` | ZisK intentionally or accidentally implements less than the RV64IM Sail behavior for an in-scope opcode.                   | Prove compliance on the complement of a precise defect predicate.                                                                                        |
| `circuit-soundness`       | A malicious witness can satisfy the constraints while disagreeing with the intended execution relation.                    | Do not advertise an unqualified compliance theorem for affected cases. Either prove a precise exclusion theorem or mark the claim blocked.               |
| `trust-shape`             | An axiom states an opcode-level conclusion that should instead be derived from a shared trust boundary plus finite proofs. | Replace with a shared boundary and derived projection theorems. The defect is closed only when the bad axiom disappears from the global theorem closure. |
| `modeling-gap`            | The Lean model deliberately abstracts something required for the real implementation claim.                                | Either move it to the scope document if it is out of scope, or express it as an explicit theorem hypothesis.                                             |

| `condition`            | A malicious witness can select signed-multiply table rows where the product-sign witness is not `na XOR nb`; the carry-chain then describes an absolute/product-shape relation rather than the intended two's-complement signed result. For ops `180`/`181`/`179` (`MUL`/`MULH`/`MULHSU`), `ZiskFv.AirsClean.ArithTableProjections.Mul.{mul,mulh,mulhsu}_np_xor_or_zero_product_shape` prove every lookup-aware row has either `np = na XOR nb` (honest) or one of the exact exceptional shapes `(na=1, nb=0, np=0)` / `(na=0, nb=1, np=0)`. The exceptional shapes pin `np = 0` while `na XOR nb = 1`, breaking the carry-chain identity the honest proof relies on; this is the witness behind the demonstrated `MUL(-1, 1) = 1` forge. The narrowed predicate `Defects.MaliciousSignedMulWitnessShape (.mul/.mulh/.mulhsu ...)` is exactly this disjunction. **Sign-range residual (MULH/MULHSU only):** the high-half signed proof additionally needs `na = MSB(op1)` and `nb = MSB(op2)` to bridge the AIR's unsigned chunk packings to the signed operands `op.toInt = packed4 − sign·2^64`. The real ZisK ArithMul circuit ENFORCES this via the indexed `range_ab`-keyed POS/NEG range lookup on `a[3]` (`zisk/state-machines/arith/pil/arith.pil:286/289/303`; honest side `arith_operation.rs:487`). The FV extraction collapses that indexed lookup to the FULL `rangeTable16` (`< 2^16`) and does not compose `ArithRangeTable` into balance, so `na=MSB`/`nb=MSB` is UNPROVABLE in-model. It is therefore CARRIED as an explicit hypothesis (`h_sign_a`/`h_sign_b`, encoded `(v.na r_a).val = if 2^63 ≤ packed4 a-chunks then 1 else 0`) on the canonical `equiv_MULH`/`equiv_MULHSU`, NOT an axiom. This is an extraction-fidelity residual (same class as the aeneas-decode / memory-timeline residuals): satisfiable for every real trace, dischargeable-in-principle by composing the indexed `ArithRangeTable` lookup into balance. MULHSU's op2 is unsigned, so the table pins `nb = 0` and only `na=MSB` (`h_sign_a`) is carried.                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `evidence`             | Clean finite-table counterexamples show `MULH` and `MULHSU` rows with `na = 1`, opposite unsigned/nonnegative operand sign, and `np = 0`. The range-table shape admits positive `d3` for that branch, so the signed high-half proof cannot recover Sail semantics from the old `np_xor` shortcut. Executable repro: a separate ZisK worktree at commit `0142ab5d7` was branched as `repro/mulh-mulhsu-malicious-witness-demo` and run on 2026-05-22 with Docker image `zisk-arith-mul-repro:mulh-demo`. The bad row family uses `na = 1`, `nb = 0`, `np = 0`, `c = 1`, `d = 0`, carries `[-1,-1,-1,-1,0,0,0]`; for high-half cases this contradicts Sail, which returns `0xffffffffffffffff`, not `0`. |
| `claim impact`         | The global theorem takes `h_known_bugs : Defects.NoKnownDefect env`. All three of `MUL`, `MULH`, `MULHSU` now carry a **narrowed, non-vacuous** `h_not_forge : ¬ ((na=1 ∧ nb=0 ∧ np=0) ∨ (na=0 ∧ nb=1 ∧ np=0))` binder (the unfolded `¬ Defects.MaliciousSignedMulWitnessShape (.mul/.mulh/.mulhsu ...)`); the global dispatcher supplies it from `NoKnownDefect`. Non-vacuity is gate-checked by `Defects.honest_mul_witness_not_malicious` / `honest_mulh_witness_not_malicious` / `honest_mulhsu_witness_not_malicious`. MULH/MULHSU additionally carry the SIGN-RANGE RESIDUAL `h_sign_a` (and `h_sign_b` for MULH) on `ZiskFv.Equivalence.{MulH,MulHSU}.equiv_{MULH,MULHSU}` — caller-supplied hypotheses, NOT axioms (the per-theorem `Lean.collectAxioms` closure in `baseline-equiv-axiom-deps.txt` is UNCHANGED: 0 `ZiskFv.*` axioms, identical to `equiv_MUL`).
