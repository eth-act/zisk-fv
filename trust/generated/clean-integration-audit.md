## Dispatch And Surface Uniformity

postcondition for its arms and `True` for every other arm; this is manually maintained in
`ZiskFv/Compliance.lean` and `ZiskFv/Compliance/Dispatch/*.lean`.

- Family dispatcher conclusions in `OpEnvelope.exec_eq`: `exec_eq_branch`, `exec_eq_nomem`, `exec_eq_rtype_binary`, `exec_eq_itype_binary`, `exec_eq_shift`, `exec_eq_add_rtypew`, `exec_eq_ldsd`, `exec_eq_divu`, `exec_eq_misc`, `exec_eq_remaining`.

  so a dispatcher call to a canonical `Equivalence.equiv_<OP>` still reaches the wrapper layer.
- All active dispatchers call canonical `ZiskFv.Equivalence.<Op>.equiv_<OP>` targets.

## Uniformity Conclusions

- Clean static-provider routes (`Binary.staticLookupComponent` and
  `BinaryExtension.shiftStaticLookupComponent`) are the cleanest integrated shape:
  caller supplies a concrete provider table row and soundness uses `table.Spec`, not
  component completeness.
- ArithDiv, ArithMul, MemAlignByte, and MemAlignReadByte no longer pull
  Clean completeness into soundness closure; soundness uses direct row/table-spec
  projections, while component/via_component paths remain helper infrastructure.
- Memory load/store proofs use full-ensemble witness constructors. That is a real Clean
  integration path, and it is architecturally different from the static-provider rows.
- `BinaryAdd` and full Main/ensemble components are built and balanced in source but are not
  the current global theorem route. They are scaffold/helper paths, not evidence that the
  global theorem is incomplete.
- The dispatcher layer is manually partitioned into ten `exec_eq_*` families with `True`
  fallthroughs. This is sound when all families are maintained correctly, but it is not a
  uniform single-source dispatch architecture.
- The intended public theorem surface is the global compliance theorem plus the 63
  canonical `ZiskFv.Equivalence.<Op>.equiv_<OP>` theorems. Wrapper and EquivCore
  routes are implementation details.

## Scaffold And Coverage Questions

- `BinaryAdd` has real Clean component and `via_component` lemmas, but canonical `ADD`/`ADDI` use `Binary.staticLookupComponent`; review whether BinaryAdd is dead scaffold, future scaffold, or an accepted provider path missing from `OpEnvelope`.
- Full-ensemble/Main-Mem helpers construct memory witnesses, but the global theorem remains opcode-envelope driven; review whether this is the intended stable architecture or a staged migration.
- Signed `MUL`/`MULH`/`MULHSU` and signed `DIV`/`DIVW`/`REM`/`REMW` are defect-gated, so provider classification there describes the visible surface rather than a completed proof route.
- Global soundness closure now contains zero Clean completeness declarations; new leaks are blocked by `trust/scripts/check-clean-integration.sh`.
