# Structural-unpacking exceptions to the anti-laundering metric
#
# Per the anti-laundering principle in CLAUDE.md, every refactor must
# reduce or hold the per-theorem `total=` / `hypothesis=` counts in
# `trust/baseline-hypothesis-count.txt`. This file lists the narrow
# exception class: opcodes where the discharge takes the form of
# *structural unpacking* — replacing a single compressed *promise
# hypothesis* (such as `h_byte_sum_circuit` asserting the full
# Sail-form opcode output) with the explicit validator + universal-
# row-constraint + structural-pin parameters needed to derive it from
# the trust ledger.
#
# Per-opcode metric apparently grows (the new parameters outnumber the
# single removed binder), but the **global-theorem trust footprint
# provably does not**: those added parameters are exactly the shape
# `Compliance.lean` already takes ONCE per provider AIR, not per
# opcode. The structural-unpacking refactor is therefore a real trust
# reduction; the metric is fooled by per-opcode counting.
#
# Format (one block per opcode):
#   OPCODE: <equiv name>
#   DROPPED: <binder name(s)>
#   ADDED: <binder name(s) added by structural unpacking>
#   RATIONALE: <one-paragraph why this isn't laundering>
#   PIL-BASIS: <citations to PIL constraints justifying the unpacking>
#
# Reviewers: when approving a baseline regen for an opcode in this
# list, confirm that (a) the DROPPED binder matches what's listed
# here, (b) the ADDED binders are exactly the validator / row-
# constraint / structural pin shapes — NOT new promise hypotheses,
# (c) the eventual `Compliance.lean` will collapse the ADDED binders
# into shared parameters.
#
# Adding a new entry requires CODEOWNER review.

#   e2.x0.val + e2.x1.val * 256 + ... + e2.x7.val * 2^56

#

# ~14 added parameters are exactly the structural-unpacking shape:
# each is either a `Valid_<AIR>` instance, a universal-over-rows
# constraint, or a transpile-derived pin — none is a new promise
# hypothesis. In `Compliance.lean` the validator and chain-holds
# parameters are taken ONCE per provider AIR (shared across all
# Arith opcodes), and the mode pins follow from `transpile_<OP>`'s
# row-shape contract.
#
# PIL-BASIS for all 9 entries below:
#   * arith.pil:280 — `arith_range_table_assumes(ARITH_RANGE_CARRY, carry[index])`

#     bound by row-level constraints
#   * arith.pil:340-360 — packing of c-chunks / d-chunks into the
#     bus-emission tuple (corresponds to h_byte_lo / h_byte_hi)
