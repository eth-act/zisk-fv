the question

What are you trusting when a ZisK proof verifies?

A ZKVM proof says: "this RISC-V program ran correctly on this circuit." But the circuit is just polynomial constraints; "ran correctly" means "matches the RISC-V architectural spec." Without a formal proof tying the two together, the claim is a wish. zisk-fv is that proof — the global theorem ZiskFv.Equivalence.Compliance.Global.zisk_riscv_compliant_program_bus dispatches all 63 RV64IM opcodes to the Sail spec, and its transitive axiom closure (72 axioms, 10 classes) is mechanically asserted equal to the ledger on every PR. This page lays out exactly what those axioms say.

What we trust, exactly — 72 axioms, 10 classes

The global compliance theorem's transitive #print axioms closure (recorded in trust/baseline-zisk-riscv-compliant.txt and hashed line-by-line in trust/baseline-axioms.txt) is the trusted computing base. The V2 gate's check-closure-vs-baseline asserts they match exactly on every PR. Counts and locations refresh against the current ledger (docs/fv/trusted-base.md).

Inherited The RISC-V specification itself — 1 source, machine-translated, not counted in the 72

We don't prove that the Sail-RISC-V model is correct — that's an inherited trust from the formally-developed Sail spec used by Nethermind / OpenLabs and the broader community. Every opcode's "intended behavior" is what the Sail model says it is. We use the stock NethermindEth/sail-riscv-lean transpiler output unchanged: the LeanRV64D Lean module is mechanically generated from the upstream Sail RV64 spec — RV64 is native there, no widening on our side, no fork. The Sail compiler + sail-riscv source pin live in flake.lock; the build is reproduced by nix build .#sail-lean-tree.

Class #4 Bus / lookup soundness — 14 axioms · OperationBus/Bridge.lean (3) + MemoryBus/{MemBridge,MemAlignBridge}.lean (9 + 2)

ZisK's circuit is a constellation of AIRs glued together by lookup and permutation arguments over named "buses". Proving these arguments sound from first principles would mean formalizing the proving system — out of project scope. Each bus's protocol is trusted via a small set of axioms that say "the rows on bus X match across providers and consumers." Each axiom's docstring cites the PIL line and Rust transpile function it mirrors. Same trust class as openvm-fv's bus hypotheses.

The uber theorem and its closure

The verification claim is one Lean theorem in ZiskFv/Equivalence/Compliance/Global.lean: 

theorem zisk_riscv_compliant_program_bus :
  ∀ (env : OpEnvelope) (state : SystemState),
    (env-side circuit-witnessed bus_effect) = LeanRV64D.Functions.execute _ state

It dispatches all 63 RV64IM opcodes through a 35-arm OpEnvelope sum type to the per-opcode equiv_<OP>_from_trust wrappers under Compliance/<Op>Exemplar.lean (plus DivPilot.lean); each wrapper discharges its canonical equiv_<OP> theorem's promise hypotheses from the trust ledger.

The closure-vs-baseline gate (V2's check-closure-vs-baseline) computes the transitive #print axioms closure of zisk_riscv_compliant_program_bus at every CI run and asserts it equals the set of names in trust/baseline-axioms.txt exactly. Any addition (a proof silently picks up a new axiom) or removal (dead trust no longer reachable) fails the gate. This is what makes "72 axioms" not just a count but a mechanically-enforced equality.

CI-enforced — drift is mechanically caught (9 V1 + 3 V2 gates)

The trust ledger isn't just documentation: every PR runs both layers via .github/workflows/trust-gate.yml.

V1 — 9 syntactic checks (check-all.sh, no build):

  • Locality. Every axiom / opaque / constant / unsafe def / partial def / @[extern] / @[implemented_by] must live in trust/allowed-axiom-files.txt.
  • Baseline integrity. Every axiom's name, location, and source-text statement is hashed into trust/baseline-axioms.txt. Any add, remove, rename, or weakening shows up as a diff.
  • No OUTPUT-EQ leak. No canonical equiv_<OP> may take a retired hypothesis (h_rd_val, h_byte_sum, …). Enforced uniformly across all 63 opcodes; loads were closed by deriving their byte equations from circuit witnesses.
  • Floors + cross-witness. ≥ 82 axioms in baseline, ≥ 63 canonical equiv_<OP>, plus a parser cross-witness that catches a sabotaged regenerate or an emptied allowlist.
  • Zero sorry. Under ZiskFv/{Fundamentals,Airs,Circuit,Equivalence,Tactics,Sail}.
  • Uniformity. All 63 RV64IM opcodes ship a canonical equiv_<OP>.
  • Anti-laundering tripwires. Per-theorem hypothesis count + canonical caller-burden ledger + wrapper caller-burden ledger — three independent baselines, each refusing silent drift.

V2 — 3 semantic checks (check-all-semantic.sh, requires lake build):

  • Per-theorem axiom closure. baseline-equiv-axiom-deps.txt records the transitive non-kernel axiom closure of every canonical equiv_<OP>; any silent change (additions OR removals) fails.
  • Forbidden binder types. forallTelescope + whnfR walks each canonical theorem's binders; catches the abbrev-aliasing dodge V1's regex misses.
  • Closure vs baseline. The uber theorem's transitive project-axiom closure must equal baseline-axioms.txt exactly — catches dead trust the per-theorem baseline cannot see.

Files that hold the trust surface and the gate's own configuration are listed in .github/CODEOWNERS; merging changes to any of them requires a CODEOWNER review. The trust/baseline-axioms.txt diff IS the audit surface — a reviewer reads it directly to see what changed.

how the proof is structured

Three layers, two languages, one theorem shape

Each per-opcode proof is a small composition: take the Sail pure spec, take the named-column AIR layer, plug them into the right bus model, and discharge the residual via a per-archetype discharge lemma. The toolkits below are what made the discharge mechanical.

Spec side

The Sail RV64 spec is mechanically translated to Lean by NethermindEth/sail-riscv-lean, producing the LeanRV64D module. RV64 is native there — we don't widen the spec ourselves. Our ZiskFv/RV64D/<op>.lean files are hand-written cleaner reformulations (execute_<OP>_pure) for proof ergonomics; each ships an equivalence lemma execute_<OP>_pure_equiv proving it equals LeanRV64D.Functions.execute. Any drift is a build failure — no untracked widening trust.

Circuit side

Extraction/ renders zisk.pilout into raw Lean constraint structures. Airs/ wraps those into Valid_<AIR> records with named-column accessors and per-constraint bridge lemmas. The Main AIR multiplexes opcodes; secondary AIRs (Arith, Binary, Mem, MemAlign) handle heavy ops.

Composition

Circuit/<Op>.lean proves: "if the relevant AIR rows validate and the bus carries the right entry, the Sail spec output equals the bus-effect output." Equivalence/<Op>.lean wraps that into the canonical equiv_<OP> shape; Equivalence/Compliance/<Op>Exemplar.lean wraps that into equiv_<OP>_from_trust, discharging promise hypotheses from the trust ledger; finally Compliance/Global.lean dispatches all 63 opcodes through a 35-arm OpEnvelope to produce the uber theorem.

Toolkits (Lean infrastructure)

PackedBitVec/{NoWrap, MulNoWrap, SignedNoWrap, WidePCNoWrap, Extensions, Signed}.lean — generic FGL ↔ BitVec 64 lifts handling additive carry chains, multiplicative chunks, four-quadrant signed witnesses, and PC-wide values that exceed the Goldilocks prime. None of these are trust items; all are proven.

a note on what could go wrong

What this doesn't prove

Honest scope. Three things this project deliberately doesn't cover, each with a reason.

The proving system itself

PLONK / plookup / logUp / permutation argument soundness as Lean theorems is out of scope. The bus-protocol axioms (BT-B, BT-X, MB-*) trust this layer at the bus-entry level. Closing it would mean formalizing the proving system itself.

The transpiler implementation

We axiomatize the contract that riscv2zisk_context.rs obeys, but we don't prove the Rust code matches it. A separate Rust-side verification effort would close this gap.

The Sail spec's correctness

The Sail RISC-V model is taken as ground truth, and we depend on the stock sail_to_lean output (NethermindEth/sail-riscv-lean :: LeanRV64D) at a pinned upstream commit — no fork. If a bug exists in Sail or its Lean translation, this project inherits it. Sail is widely audited by the RISC-V community, so this is the conventional bar.

two views of the same trust

Why the encodings differ — and what that costs

Both projects ultimately trust the same kinds of things (Sail, the transpiler contract, the proving system's bus protocols). What's different is whether that trust is named and tallied, or absorbed into hypothesis structures.

openvm-fv's approach

Trust as hypotheses. Each theorem assumes wf_propertiesToAssumePerRow; the trust content lives inside that predicate's structure plus the body of the transpile_op Lean function.

Pro: #print axioms on any theorem returns only Lean kernel + Mathlib axioms. Visually clean. Reviewers see "zero project-specific axioms" and that's true at the language level.

Con: Trust is diffuse — it's distributed across hypothesis predicates and a 7,470-line Lean function. Auditing requires reading the structures and reasoning about whether the implicit forward contract matches the Rust transpiler.

zisk-fv's approach

Trust as named ledger. Each trust assumption is an explicit axiom declaration with a docstring, an entry in docs/fv/trusted-base.md, and a reference in CLAUDE.md's trusted-surface section.

Pro: Every trust item is greppable, individually auditable, and load-bearing — the V2 check-closure-vs-baseline gate asserts the uber theorem's transitive axiom closure equals the ledger exactly, so dead trust is impossible by construction. The 72-axiom ledger (10 classes) is the audit artifact.

Con: #print axioms on a final theorem prints a long list, including structural bus, lookup, range, platform, and transpiler-bridge axioms. The trust ledger is more honest but also more verbose.

Practical takeaway

The two encodings carry roughly the same trust content. openvm-fv minimizes axiom count for review optics; zisk-fv maximizes ledger explicitness for trust auditing, and the V2 check-closure-vs-baseline gate raises that to a mechanical equality — the uber theorem's transitive axiom closure must equal the ledger exactly. Either project would close the remaining gap by formally verifying its Rust transpiler against its Lean contract — that follow-on Rust-side proof is open in both.

what we adopted directly

From openvm-fv: patterns we kept

Building on openvm-fv was a deliberate choice; the two projects share more than they diverge on. Here's what transferred cleanly.

Five-layer pipeline

Extraction/ (auto-generated raw constraints) → Airs/ (named-column structures) → Constraints/ (simp bridges) → Circuit/ (circuit→semantic theorems; Spec/ in openvm-fv) → Equivalence/ (per-opcode final theorem).

Final theorem shape

execute_instruction OP state = (bus_effect exec_row mem_row state).2 — the canonical equivalence. Both projects converge here.

Lookup-table trust pattern

openvm-fv's BitwiseBusEntryInstance.wf_properties is the direct template for zisk-fv's BT-B / BT-X. Trust the lookup table at the bus-entry definition site, not in proofs of XOR / AND / shift correctness.

Per-opcode equivalence layout

One file per opcode under Equivalence/<Op>.lean; pure Sail spec under Sail/<op>.lean (openvm-fv: RV32D/<op>.lean); circuit-correctness lemma under Circuit/<Op>.lean (openvm-fv: Spec/<Op>.lean). Same role-of-file.
